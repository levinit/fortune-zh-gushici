# AI 協作入口

本目錄是本專案的 agent harness：把 AI 的行為收束成可檢查、可重複、可回滾的流程。若任務涉及新增、刪改或校對作品條目，必須選用下方 workflow，不可只憑直覺直接修改 `gushici.yaml` 或 `data/`。

## Harness 原則

- **先觀察，後修改。** 開始改動前先讀 `README.md`、`正字.md`、`Makefile`、`build.py` 與相關 YAML 條目。
- **單一事實源。** `gushici.yaml` 是唯一手工維護的源數據；`data/gushici-cht`、`data/gushici-chs` 與 `.dat` 文件均由 `make compile` 生成。
- **證據優先。** 新增作品前先確認作品身份、作者、朝代、正文與常見題名；不確定時先向用戶索取來源或說明待確認點。
- **查重是閘門。** 任何新增條目前，都要用題名、作者、起句、別名與正文片段查重；已存在則報告並終止。
- **先確認再落盤。** 查重後若確定不存在，先向用戶展示擬新增的 YAML 欄位與正文摘要，取得同意後再修改。
- **繁體手工維護。** `gushici.yaml` 必須維護繁體文本並參照 `正字.md`；不得用 OpenCC 由簡體反推繁體。
- **生成不可手寫。** 不直接手工改 `data/`；修改源文件後執行 `make compile`。
- **驗證要閉環。** 每次改動後至少執行格式、生成、繁簡與 fortune smoke check 中適用的檢查，並在回覆中列出結果。
- **小步提交。** 讓每次變更只對應一個清楚意圖；不要順手整理無關內容。

## 可用 Workflow

- [新增作品條目](workflows/add-work.md)：當用戶提供一首或一組作品，希望加入 `gushici.yaml` 時使用。

## 條目規範速記

- `dynasty`：朝代字串；未知時用 `_`。
- `authors`：作者列表；多人作品或傳為他人回贈時按作品實際作者分條目。
- `title`：題名列表，生成時以 `·` 連接。後續元素可放分題、序號或通行小題，不硬取正文首句。
- `body`：正文，使用繁體中文；原文小序、題序也放入正文；段落與分題用空行分隔，不含單獨一行 `%`。
- `alias`：又名，字串或列表；生成時顯示「又名：...」。
- `notes`：說明文字列表；不要使用單數 `note`。
- `group`：只用於把彼此獨立但需要連在一起輸出的作品分組；分組鍵不必受作者限制，但應穩定、可讀、能避免誤合併。
- `suffix`：署名後綴，如「（節選）」。

常見判斷：

- 組詩或組文通常存為一個條目，分題與序號寫入 `body`，必要時用 `notes` 說明收錄範圍。
- 原文自帶的小序、題序屬於正文，放在 `body`；編者補充、版本疑義、收錄範圍才放 `notes`。
- 唱和、前後作、回贈等彼此獨立的作品分成多個條目；需要相鄰輸出時使用同一 `group`。
- 詞牌、曲牌相同但無正式小題時，仍只保留詞牌或曲牌本身；不要為了辨識而硬取正文首句加入 `title`。
- 繁體正字疑問先查 `正字.md`；OpenCC 只用於繁轉簡生成，不用於簡轉繁維護源文件。
