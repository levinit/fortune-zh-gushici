# Workflow: 新增作品條目

當用戶給出作品相關資訊並希望加入數據庫時，使用本流程。目標是穩定完成「識別作品 -> 查重 -> 用戶確認 -> 寫入 YAML -> 生成派生文件 -> 驗證」。

## 1. 收集與識別

先整理用戶提供的資訊：

- 題名、別名、詞牌或曲牌。
- 作者、朝代、可能的歸屬爭議。
- 正文、節選範圍、分題或序號。
- 背景說明、唱和關係、前後作、回贈關係。
- 來源或版本依據。

若資訊不足以可靠識別作品，不要改文件。先向用戶確認缺失項，或說明需要查證的點。若正文來自簡體，必須手工核對繁體與正字，不得直接用 OpenCC 反向轉換後寫入源文件。

## 2. 查重閘門

在任何修改前查重。至少使用下列線索中的多個：

```bash
make check KEYWORD=題名
make check KEYWORD=作者
python3 build.py --check "起句或正文關鍵句"
rg -n "題名|別名|起句|作者" gushici.yaml
```

查重時同時考慮：

- 繁簡差異與異體字。
- `title` 後續元素中的起句、小題、序號。
- `alias`、`notes`、`group`。
- 作品是否已收在某個組詩或組文的 `body` 中。

若已存在或高度疑似已存在，停止修改，回覆用戶：

- 現有條目的作者、題名、位置。
- 是完整收錄、節選收錄，還是已收在組詩/組文中。
- 若用戶真正想做的是補正文、補注或改題名，請另按刪改任務處理。

## 3. 擬定 YAML 條目

確認不存在後，先擬定條目，不要立刻寫入。向用戶展示關鍵欄位並詢問是否加入。

欄位規則：

- `dynasty` 使用繁體朝代名；未知時用 `_`。
- `authors` 使用列表，如 `["李白"]`。
- `title` 必須是列表：
  - 普通作品：`["靜夜思"]`
  - 分題或序號：`["秋興八首", "其一"]`
  - 詞牌/曲牌加通行小題：`["水龍吟", "登建康賞心亭"]`
  - 不要為了區分同詞牌而硬取正文首句。
  - 不要另設 `incipit` 或 `first_line`。
- `alias` 用於又名，如 `alias: "遊玄都觀"`；多個又名使用列表。
- `notes` 必須是列表，用於編者補充、版本疑義、收錄範圍。
- `group` 僅用於多個獨立作品需要合併輸出；同一組作品使用完全相同的 group key。
- `suffix` 用於「（節選）」等署名後綴。
- `body` 保留正文換行；原文小序、題序放在正文開頭；組詩分題、序號放在正文內，分題與內容之間用空行。

分類規則：

- 組詩：通常一個 YAML 條目，`title` 為總題，`body` 中列出「其一」「其二」與各首正文。
- 唱和/回贈：彼此獨立，分成多個 YAML 條目；需要連續輸出時用 `group`。
- 詞牌/曲牌：若有正式小題則保留；若沒有正式小題，不要硬取正文首句作為 `title` 第二元素。
- 普通重名詩題：不要自動加起句，先判斷是否真有辨識必要。
- 小序/題序：若是原文自帶，放入 `body`；若是編者補充，放入 `notes`。

建議確認話術：

```text
我查重後未發現已有條目。擬新增：

- 朝代：
- 作者：
- 題名：
- 又名：
- 說明：
- 正文首句：

是否按這個結構加入？
```

## 4. 寫入源文件

取得用戶同意後再修改 `gushici.yaml`。

- 優先放在同朝代、同作者附近，維持現有人工順序。
- 不為了新增一首作品重排整個文件。
- 只改本任務相關內容；不要順手改無關字詞。
- 手工編輯使用 patch，保持 YAML 縮進與列表格式一致。

## 5. 生成派生文件

修改 `gushici.yaml` 後執行：

```bash
make compile
```

這會生成：

- `data/gushici-cht`
- `data/gushici-cht.dat`
- `data/gushici-chs`
- `data/gushici-chs.dat`

不要跳過 `.dat`，也不要手工改 `data/`。

## 6. 驗證

至少執行：

```bash
python3 -m py_compile build.py
git diff --check
python3 build.py --check "題名或起句"
fortune data/gushici-cht
fortune data/gushici-chs
```

若本機有 OpenCC，建議核對簡體文件確由繁體派生：

```bash
opencc -i data/gushici-cht -o /private/tmp/gushici-chs.check -c t2s.json
diff -u data/gushici-chs /private/tmp/gushici-chs.check
```

驗證失敗時先修復，不要把半生成狀態交給用戶。

## 7. 回覆用戶

完成後簡要回覆：

- 新增了哪個條目，位於哪個文件。
- 生成了哪些派生文件。
- 執行了哪些驗證。
- 若未提交，明確說「尚未提交」；若用戶要求提交，再提交。

若流程在查重階段終止，回覆應以「已存在」為主，不做文件改動。
